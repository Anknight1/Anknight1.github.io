第二十一篇：实战四：手搓Human in the Loop

前言

本文是「大模型学习」系列文章之一，持续更新中。

前几篇实战中，Agent 都是完全自主运行的。但在生产环境中，有些操作风险极高——比如删除数据库记录、发送邮件、调用付费接口——我们希望 Agent 在执行前先征得人工确认。

本篇实现一个完整的 Human-in-the-Loop（HITL）Agent，核心能力：

  • 执行前自动识别高风险工具调用
  • 暂停执行，等待人工审批（Approve / Reject / Edit）
  • 状态持久化，支持跨请求恢复
  • 完整的 REST API 交互界面

总结

本篇实现了一个完整的 Human-in-the-Loop Agent，核心设计要点：

风险分级：将工具调用分为低/中/高三个风险等级，只对中高风险操作触发人工审批，避免过度打扰。

状态持久化：通过 Checkpointer 保存会话状态，支持跨请求的中断与恢复，Agent 可以无限期等待人工输入。

三种审批决定：批准（按原参数执行）、编辑（修改参数后执行）、拒绝（终止该操作），给审批人充分的控制权。

REST API 解耦：启动、查询、审批通过独立接口完成，便于集成到任何前端或审批系统。

HITL 是生产级 Agent 系统的重要安全机制，在自主性与可控性之间取得平衡。
