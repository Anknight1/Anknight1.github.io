五、Spring AI Alibaba 原生长期记忆方案

除了接入 Mem0，Spring AI Alibaba 还提供了一套原生的长期记忆方案，基于 MemoryStore + ReactAgent 实现。

5.1 核心概念

Spring AI Alibaba 将记忆分为两层：

短期记忆（Short-term Memory）：存储当前会话的对话历史，通过 MemorySaver（checkpointer）持久化，以 threadId 为键。

长期记忆（Long-term Memory）：存储跨会话的持久化数据（用户画像、偏好等），通过 MemoryStore 持久化，以 namespace + key 为键。

六、两种方案对比

对比维度 Mem0 方案 Spring AI Alibaba 原生方案
记忆智能化 高（LLM 驱动的事实提取和去重） 低（手动管理，无自动提取）
部署复杂度 高（需要 Python 服务 + pgvector + Neo4j） 低（纯 Java，无额外依赖）
性能 较慢（每次对话需调用 LLM 提取事实） 快（直接读写存储）
记忆质量 高（自动去重、更新、删除过时记忆） 取决于业务逻辑实现
适用场景 需要智能记忆管理的复杂 Agent 简单的用户画像存储和注入
图关系支持 支持（Neo4j 知识图谱） 不支持
生产就绪 需要额外运维 Mem0 服务 可直接使用数据库存储

选型建议：

如果你的 Agent 需要处理复杂的用户信息更新（比如用户换了工作、搬了城市），并且希望记忆能够自动去重和更新，推荐使用 Mem0 方案。如果只是简单地存储和读取用户偏好，Spring AI Alibaba 原生方案更轻量，部署更简单。

七、常见问题

Q：接入 Mem0 后响应速度很慢怎么办？

这是 Mem0 的已知问题，因为每次对话都需要调用 LLM 进行事实提取，耗时较长（可能超过 1 分钟）。可以考虑：异步写入记忆（先返回响应，后台异步存储）；或者只在对话结束时批量写入，而不是每轮都写入。

Q：Mem0 的 LLM 配置为什么不能用 Qwen？

目前 Qwen 在 Mem0 的 CRUD 决策环节存在 JSON 结构解析问题，建议使用 DeepSeek 或 OpenAI 作为 Mem0 内部的 LLM。

总结

本文介绍了在 Spring AI Alibaba 中接入 Mem0 的完整流程：

首先通过 Docker Compose 启动 Mem0 服务（包含 pgvector 和 Neo4j），然后在 Spring Boot 项目中引入 spring-ai-alibaba-starter-memory-mem0 依赖，通过 application.yml 配置 Mem0 的连接信息和内部 AI 模型，最后使用 Mem0ChatMemoryAdvisor 自动管理对话记忆的读写。

同时也介绍了 Spring AI Alibaba 原生的长期记忆方案（MemoryStore + ModelHook），并对两种方案进行了对比，帮助读者根据实际需求选择合适的方案。
