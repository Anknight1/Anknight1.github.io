第二篇：Skill的渐进式披露

发表于2026-04-29|更新于2026-04-29|大模型学习8-Skills（智能体技能）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

上一篇介绍了 Agent Skills 的基本概念。本篇深入讲解 Skills 最核心的设计机制——渐进式披露（Progressive Disclosure），理解它是如何让数百个 Skills 同时可用而不压垮上下文窗口的。

正文

一、问题背景：上下文窗口的限制

假设你有 100 个 Skills，每个 SKILL.md 文件包含约 5000 个 tokens 的详细说明。如果把所有 Skills 都加载到上下文中：

即使强行限制，最多也只能同时加载 40 个 Skills（200,000 / 5000 = 40），这给 Skills 的使用带来了严重的扩展性问题：用户必须手动管理哪些 Skills 处于激活状态，频繁切换带来大量摩擦。

二、渐进式披露的核心思想

渐进式披露的核心原则只有一句话：只加载当时需要的内容。

这个思想来自 UX 设计领域——先展示基本信息，用户有需求时再展示详细功能。在 Agent Skills 中，这个概念被迁移到了认知层面：先让 AI 知道有哪些能力可用（元数据），当真正需要某个能力时再加载完整的执行指令（正文）。

用一个生活类比来理解：传统做法是新员工入职第一天就把公司所有流程文档、规章制度、操作手册全部打印出来堆在桌上。渐进式披露的做法是：先给一份简短的岗位说明，等遇到具体问题时，再告诉他去翻哪本手册的哪一页。

六、渐进式披露与上下文工程的关系

渐进式披露本质上是**上下文工程（Context Engineering）**的一种具体实践。

上下文工程的核心目标是：在有限的上下文窗口内，放入最有价值的信息。渐进式披露通过三层架构实现了这个目标：

  • 筛选（Filter）：通过相关性评估，只加载与当前任务相关的 Skills
  • 持久化（Persist）：元数据始终常驻，确保 AI 知道有哪些能力可用
  • 隔离（Isolate）：不同 Skills 的内容相互隔离，按需加载，避免干扰

这与 Manus 等先进 Agent 系统的上下文管理思路是一致的：不是把所有信息都塞进上下文，而是精心管理什么时候加载什么信息。

总结

渐进式披露是 Agent Skills 最核心的设计机制，它通过三层加载架构解决了上下文窗口限制的问题：

第一层元数据（约 100 tokens/Skill）始终常驻，让 AI 知道有哪些能力可用；第二层正文指令（约 5000 tokens/Skill）在触发时按需加载；第三层附加资源仅在真正需要时才加载。

这种设计将 Skills 的实际可用数量从 40 个提升到了 1500 个，响应延迟降低了 93%，同时还支持多个 Skills 自动组合协作，无需开发者手动管理。

理解渐进式披露，不仅能帮助我们更好地使用 Skills，也能指导我们设计出更高质量的 SKILL.md 文件——把最关键的触发信息放在元数据层，把详细的执行 SOP 放在正文层，把不常用的参考资料放在资源层。
