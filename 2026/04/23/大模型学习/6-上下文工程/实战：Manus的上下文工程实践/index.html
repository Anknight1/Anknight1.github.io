第四篇：实战：Manus的上下文工程实践

发表于2026-04-23|更新于2026-04-29|大模型学习6-上下文工程
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

2025 年 7 月，Manus 首席科学家 Yichao ‘Peak’ Ji（季逸超）发表了一篇技术博客：《AI Agent 的上下文工程：构建 Manus 的经验教训》。这篇文章来自真实的生产环境，是 Manus 团队在服务数百万用户过程中积累的”战地笔记”。

Manus 是一个面向通用任务的 AI Agent 产品，平均每次任务需要约 50 次工具调用，输入/输出 Token 比例约为 100:1。在这种高强度的 Agent 场景下，上下文工程的重要性远超模型选择本身。

本文将深入解析 Manus 团队总结的 6 大上下文工程实践。

正文

一、核心选择：上下文学习 vs 端到端训练

在 Manus 项目启动之初，团队面临一个关键抉择：

  • 方案 A：使用开源基础模型，训练一个端到端的 Agent 模型
  • 方案 B：基于前沿模型的 in-context learning 能力构建 Agent

季逸超选择了方案 B，理由很务实：

“端到端训练每次迭代需要数周时间。对于快速迭代的应用，特别是在 PMF 之前，这样缓慢的反馈循环是致命的。”

押注上下文工程的好处：

  • 改进可以在几小时而不是几周内发布
  • 产品与底层模型正交——模型升级时，Agent 自动受益
  • 如果模型进步是涨潮，Manus 是船，而不是固定在海床上的柱子

这个选择背后有一个重要洞察：上下文工程的价值，在当前阶段远大于模型微调。

二、实践一：围绕 KV-Cache 设计

核心指标：KV-Cache 命中率

季逸超认为，KV-Cache 命中率是生产阶段 AI Agent 最重要的单一指标，直接影响延迟和成本。

为什么 KV-Cache 如此重要？

在 Agent 的每次迭代中，上下文会不断追加新内容（action + observation），但前缀部分（System Prompt + 历史记录）是不变的。如果这部分能命中 KV-Cache，就不需要重新计算，可以大幅降低成本和延迟。

以 Claude Sonnet 为例：

  • 缓存命中的输入 Token：$0.30 / MTok
  • 未缓存的输入 Token：$3.00 / MTok
  • 相差 10 倍！

如何提高 KV-Cache 命中率？

1. 保持 Prompt 前缀稳定

2. 保持 Context 只追加（Append-only）

3. 确保序列化的确定性

三、实践二：掩码而非移除工具

问题：工具数量爆炸

随着 Agent 功能增加，工具数量会急剧增长。特别是 MCP 协议普及后，用户可能插入数百个工具，导致模型选择错误工具的概率大增。

直觉方案：动态加载工具（类似 RAG）

Manus 团队也尝试过按需动态添加/移除工具，但发现了两个严重问题：

  1. 破坏 KV-Cache：工具定义通常在 System Prompt 前后，任何修改都会使后续所有内容的缓存失效
  2. 模型困惑：历史记录中引用了已被移除的工具，模型会产生幻觉或 Schema 违规

Manus 的解决方案：状态机 + Token Logits 掩码

四、实践三：将文件系统用作无限 Context

问题：Context 窗口不够用

即使是 128K Token 的上下文窗口，在真实 Agent 场景中也经常不够用：

  • 网页内容、PDF 文档可能非常大
  • 超过某个长度后，模型性能会下降
  • 长输入成本高昂

Manus 的解决方案：文件系统 = 无限 Context

五、实践四：通过背诵操纵 Attention

问题：长任务中 Agent 容易”忘记”目标

Manus 的典型任务平均需要 50 次工具调用。在如此长的执行过程中，Agent 很容易偏离最初的目标——这就是”Lost in the Middle”问题在 Agent 场景中的体现。

Manus 的解决方案：动态维护 todo.md

你可能注意到 Manus 在处理复杂任务时，会创建一个 todo.md 文件并持续更新它：

为什么这个技巧有效？

每次更新 todo.md，Agent 都会将当前任务目标背诵到 Context 的末尾。由于 LLM 对 Context 末尾的内容注意力最强（Recency Bias），这相当于在每一步都提醒模型：”你的目标是什么，你已经完成了什么，还剩什么要做”。

这是一个用自然语言操纵模型 Attention 的优雅技巧，无需任何架构改动。

六、实践五：保留错误路径

反直觉但重要的原则

当 Agent 犯错时，直觉上我们想清理错误记录、重试操作。但 Manus 团队发现：保留错误路径，往往比清除它更有效。

七、实践六：避免 Few-Shot 模式固化

问题：模型是优秀的模仿者

LLM 会模仿 Context 中的行为模式。如果 Context 中充满了类似的 action-observation 对，模型会倾向于重复这个模式，即使它不再是最优的。

八、Manus 经验的核心洞察

总结 Manus 团队的 6 大实践，可以提炼出几个核心洞察：

洞察一：避免过度工程

产品性能的最大飞跃往往来自简化架构、移除不必要的技巧，而不是增加更复杂的上下文管理技术。给予模型更多信任，让它自己决策。

洞察二：工程目标是简化

上下文工程的最终目标应是让 LLM 的工作变得更简单，而非更困难。每一个上下文管理技巧，都应该降低模型的认知负担，而不是增加它。

洞察三：可恢复性优于压缩率

在设计上下文压缩策略时，可恢复性比压缩率更重要。宁可保留更多信息（通过文件系统），也不要做不可逆的有损压缩。

洞察四：错误是资产，不是负债

在 Agent 系统中，错误记录是宝贵的学习信号。保留错误路径，让模型从失败中学习，是提升 Agent 可靠性的关键。

总结

Manus 的上下文工程实践可以用一句话概括：精心设计 Context 的形状，而不是堆砌 Context 的数量。

六大核心实践：

  1. 围绕 KV-Cache 设计：保持前缀稳定，只追加不修改，确保序列化确定性
  2. 掩码而非移除工具：用状态机 + Logits 掩码约束工具选择，避免破坏缓存
  3. 文件系统作为无限 Context：将大型内容外化到文件，保持压缩可恢复
  4. 背诵操纵 Attention：动态维护 todo.md，将目标持续推入 Context 末尾
  5. 保留错误路径：错误记录是学习信号，不要清除它
  6. 避免 Few-Shot 固化：引入结构化变化，防止模型陷入重复模式

这些经验来自真实的生产环境，是 Manus 团队通过”随机梯度下降”（Stochastic Graduate Descent）反复迭代得出的局部最优解。
