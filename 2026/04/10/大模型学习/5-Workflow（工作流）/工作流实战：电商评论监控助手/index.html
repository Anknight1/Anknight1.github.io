第十七篇：工作流实战：电商评论监控助手

|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

本文通过一个完整的实战项目——电商评论监控助手，演示如何综合运用 N8N 和 Dify 构建一个生产级的 AI 工作流应用。

项目背景

需求：电商运营团队需要实时监控各平台（淘宝、京东、拼多多等）上的商品评论，及时发现差评和负面舆情，并生成每日分析报告，辅助运营决策。

核心功能：

  • 定时抓取电商平台的商品评论数据
  • AI 分析情感倾向（正面/负面/中性）
  • 识别差评关键问题（物流、质量、客服等）
  • 生成每日评论分析报告
  • 发现高风险差评时自动告警

整体架构

技术选型

N8N：负责整体工作流编排，包括定时触发、数据抓取、数据库操作、消息通知等非 AI 部分。

Dify：负责 AI 分析部分，包括情感分析、问题分类、风险评估，通过 API 被 N8N 调用。

MySQL：存储抓取的原始评论数据和分析结果。

电商 API / 爬虫服务：数据来源，抓取各平台的商品评论。

第一步：评论抓取工具

数据获取方案

电商评论数据获取有两种方案：

方案一：电商开放平台 API（推荐）

淘宝、京东等平台提供官方 API，数据稳定可靠，但需要申请开发者账号和相应权限。

方案二：网页爬虫

通过模拟浏览器请求抓取评论数据，无需申请权限，但稳定性较差，需遵守 robots.txt 规则。

本文以方案一为主，以京东评论 API 为例介绍数据获取。

评论内容的特点

电商评论有以下特点，需要针对性处理：

  • 短文本：大多数评论较短，信息密度高
  • 表情符号：大量 emoji
  • 追评内容：追评：xxx 格式
  • 晒图说明：包含图片描述文字

工作流测试

手动触发测试：

  1. 在 N8N 界面点击「Execute Workflow」
  2. 查看每个节点的输入输出
  3. 检查数据库是否有新记录
  4. 验证告警是否正常发送

常见问题排查：

  • 评论抓取服务无法连接：检查服务是否启动，网络是否通畅
  • Dify API 返回错误：检查 API Key 是否正确，工作流是否已发布
  • MySQL 插入失败：检查字段类型是否匹配，是否有重复主键
  • 告警未发送：检查 IF 节点条件是否正确，Slack/邮件配置是否有效

总结

电商评论监控助手是一个典型的 AI + 自动化工作流应用。通过 N8N 处理数据流转和系统集成，Dify 处理 AI 分析，两者分工明确，各司其职：

  • N8N 负责定时触发、数据抓取、清洗、存储、告警等工程化工作
  • Dify 负责情感分析、问题分类、风险评估等 AI 智能分析工作
  • MySQL 负责数据持久化，支持后续的统计分析和可视化

这种架构模式可以推广到很多类似的场景，如社交媒体监控、客服工单分析、用户反馈处理等。AI Workflow 的核心思想是：让每个工具做它最擅长的事，通过合理的编排将它们串联成一个高效、可靠的自动化系统。
