第五篇：Harness Engineer是什么

发表于2026-04-24|更新于2026-04-29|大模型学习6-上下文工程
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

2025 年底，OpenAI 发布了一篇震惊业界的实验报告：3 名工程师，5 个月，完成了 100 万行代码的开发，且零人工编写代码。

这个实验的核心不是模型有多强，而是他们构建了一套让 AI Agent 可靠工作的”驾驭系统”。Mitchell Hashimoto（HashiCorp 创始人）在博客中把这套方法论称为 Harness Engineering，并坦言：”我不知道业界有没有公认的术语，我自己管这叫 harness engineering。”

几周之内，Harness 成了 AI Agent 开发圈绕不开的概念。

正文

一、什么是 Harness Engineering？

一句话定义：Agent = Model + Harness。你不是模型，那你就是 Harness。

“Harness” 的本意是马具——缰绳、马鞍、嚼子——这是一套引导强大但不可预测的动物的完整装备。Harness Engineering 借用这个比喻：LLM 就像一匹蛮力十足但方向感不太行的马，Harness Engineering 就是为它打造一套”黄金缰绳”，让它跑得又快又稳。

Harness 就是模型之外的一切：系统提示词、工具调用、文件系统、沙箱环境、编排逻辑、约束机制、反馈回路。模型本身只是能力的来源，只有通过 Harness 把状态、工具、反馈和约束串起来，它才真正变成一个可靠的 Agent。

一个形象的类比：

模型是 CPU，Harness 是操作系统。CPU 再强，OS 拉胯也白搭。你买了最新款 M5 芯片，装了个崩溃不断的系统，体验还不如老芯片配稳定的 OS。

二、Harness 与 Prompt/Context Engineering 的关系

很多人会把这三个概念混淆，但它们是嵌套关系，解决的是完全不同层面的问题：

层级 解决的核心问题 关注点 典型工作
Prompt Engineering 表达——怎么写好指令 塑造局部概率空间，让模型听懂意图 系统提示词设计、Few-shot 示例、思维链引导
Context Engineering 信息——给 Agent 看什么 确保模型在合适时机拿到正确且必要的信息 上下文管理、RAG、记忆注入、Token 优化
Harness Engineering 执行——整个系统怎么防崩、怎么量化、怎么持续运转 长链路任务中的持续正确、偏差纠正、故障恢复 文件系统、沙箱、约束执行、熵管理、反馈回路

简单任务里，提示词最重要；依赖外部知识的任务里，上下文很关键；但在长链路、可执行、低容错的真实商业场景里，Harness 才是决定成败的东西。

三、为什么瓶颈不在模型而在 Harness？

这个结论听起来反直觉，但数据支持它。

Can.ac 的实验：同一个模型，只换了文件编辑接口的调用方式，编码基准分数从 6.7% 直接跳到 68.3%。模型没变，变的是外围的那套系统。

LangChain 的实践：优化了 Agent 运行环境（文档组织方式、验证回路、追踪系统），在 Terminal Bench 2.0 上从全球第 30 名升到第 5 名，得分从 52.8% 提升到 66.5%。模型没换，Harness 换了。

常见误区：很多团队一遇到 Agent 表现不好，第一反应是”换更强的模型”或”调整提示词”。但实验证明，瓶颈大概率不在模型智能水平，而在 Harness 的基础设施质量。

四、Harness 包含哪些组件？

理解 Harness 的最好方式，是看模型做不到什么，然后用 Harness 来补：

模型做不到 Harness 怎么补 核心组件
记住多轮对话历史 维护对话历史，每次请求时拼进上下文 记忆系统
执行代码、跑命令 提供 Bash + 代码执行环境 通用执行环境
获取实时信息 Web Search、MCP 工具 外部知识获取
操作文件和环境 文件系统抽象 + Git 版本控制 文件系统
知道自己做对了没有 沙箱环境 + 测试工具 + 浏览器自动化 验证闭环
在长任务中保持连贯 上下文压缩、记忆文件、进度追踪 上下文管理

五、成熟 Harness 的六层架构

从系统设计角度看，一个成熟的 Harness 有清晰的层次结构：

可以类比成给一个新手员工搭建的完整工作环境：

  • L1 是岗位说明书（告诉 ta 该关注什么）
  • L2 是办公工具（给 ta 用什么干活）
  • L3 是标准操作流程（按什么步骤做事）
  • L4 是项目管理系统和笔记本（怎么记住做过的事）
  • L5 是质检流程（怎么检验做对了没有）
  • L6 是红线规则和应急预案（什么事绝对不能做、出了事怎么补救）

建议：不要试图一开始就搭齐六层。从 L1（信息边界）和 L6（约束与恢复）入手，这两层投入产出比最高。

六、上下文利用率的 40% 阈值

Dex Horthy 发现了一个重要现象：168K token 的上下文窗口，用到大约 40% 的时候，Agent 的输出质量就开始明显下降。

Anthropic 也遇到了类似问题，他们称之为”上下文焦虑“：当上下文快填满时，模型会变得犹豫，倾向于提前收工——哪怕任务还没做完。

Anthropic 的解决方案：Context Resets（上下文重置）

九、Harness Engineer 是什么职位？

随着 Harness Engineering 的兴起，”Harness Engineer”作为一个新兴职位开始出现在 AI 行业的招聘信息中。

Harness Engineer 的核心职责：

  • 设计和构建 AI Agent 的运行环境（文件系统、沙箱、工具集）
  • 建立约束机制和反馈回路，确保 Agent 输出的可靠性
  • 管理上下文策略，优化 Token 使用效率
  • 构建可观测性系统，量化 Agent 的性能表现
  • 设计多 Agent 协作架构

与传统 AI Engineer 的区别：

维度 AI Engineer Harness Engineer
关注点 模型选择、微调、Prompt 优化 Agent 运行环境、约束系统、反馈回路
核心技能 ML 理论、模型评估 系统设计、工程架构
工作产出 更好的模型/Prompt 更可靠的 Agent 系统
类比 发动机工程师 整车工程师

2026 年的行业趋势：

“2025 年是 Agent 之年，2026 年是 Harness 之年。模型是商品，Harness 才是护城河。” —— SaaS 领域投资人 Aakash Gupta

总结

Harness Engineering 是 AI Agent 时代的核心工程方法论，它解决的是”如何让 Agent 可靠地工作”这一根本问题。

核心公式：Agent = Model + Harness

三层嵌套关系：Prompt Engineering ⊂ Context Engineering ⊂ Harness Engineering

六层架构：信息边界层 → 工具系统层 → 执行编排层 → 记忆与状态层 → 评估与观测层 → 约束校验与恢复层

关键洞察：上下文利用率超过 40% 时，Agent 质量明显下降；瓶颈不在模型，而在 Harness 的基础设施质量。

从零开始搭建 Harness，先做 P0：创建 AGENTS.md + 自定义 Linter + 团队知识仓库化。这三件事的投入产出比最高，效果立竿见影。
