第三篇：上下文工程是什么

发表于2026-04-22|更新于2026-04-29|大模型学习6-上下文工程
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

2025 年，Cognition AI（Devin 的开发团队）的工程师提出了一个观点：

“上下文工程，实际上是构建 AI Agent 工程师的头号工作。”

Anthropic 也在其技术文档中指出：

“Agent 经常参与持续数百轮的对话，这需要仔细的上下文管理策略。”

那么，什么是上下文工程（Context Engineering）？它和我们熟悉的提示词工程有什么区别？

正文

一、定义：什么是上下文工程

上下文工程（Context Engineering） 是指通过系统性地设计、组织、筛选和动态管理 LLM 推理时的上下文信息，以最小化 Token 消耗、最大化模型输出质量的技术方法论。

LangChain 创始工程师 Lance Martin 给出了一个简洁的定义：

上下文工程 = 为 LLM 的下一步动作，精准填充所需的最小充分上下文。

这个定义有两个关键词：

  • 最小：不塞入无关信息，控制 Token 消耗
  • 充分：确保模型完成任务所需的信息都在

二、与提示词工程的区别

很多人会把上下文工程和提示词工程混淆，但它们有本质区别：

维度 提示词工程（Prompt Engineering） 上下文工程（Context Engineering）
关注点 如何写好单次输入文本 如何管理整个推理过程的信息流
时间维度 单次请求 多轮、长时间运行
适用场景 聊天机器人、单次任务 AI Agent、复杂工作流
核心技能 写作技巧、Few-shot 设计 系统架构、信息调度
主要挑战 如何让模型理解意图 如何在有限窗口内管理动态信息

一个形象的比喻：

  • 提示词工程 像是”如何向一个人提问”
  • 上下文工程 像是”如何为一个人设计完整的工作环境”

提示词工程是上下文工程的子集——系统提示词（System Prompt）的设计依然是提示词工程的范畴，但上下文工程还要处理信息的动态加载、压缩、检索和隔离。

三、上下文工程的发展阶段

上下文工程的兴起经历了三个阶段：

阶段一：提示词工程时代

随着 ChatGPT 的爆火，”如何写好 Prompt”成为热门话题。Few-shot、Chain-of-Thought、角色扮演等技巧层出不穷。这个阶段的核心是优化单次输入。

阶段二：RAG 时代

为了让 LLM 连接外部知识库，RAG（检索增强生成）应运而生。这个阶段开始关注如何将外部信息注入上下文，但仍然以单次检索为主。

阶段三：Agent + 上下文工程时代

随着 LLM 工具调用能力的成熟，AI Agent 开始执行多步骤、长时间的复杂任务。上下文不再是静态的，而是随着 Agent 的每一步操作动态增长。如何管理这个动态增长的上下文，成为构建可靠 Agent 的核心挑战。

四、上下文工程的三大核心策略

LangChain 团队将上下文工程的核心策略归纳为三类：筛选（Curate）、持久化（Persist） 和 隔离（Isolate）。

策略一：筛选上下文（Curating Context）

筛选是指控制每一轮 Agent 可见的 Token 数量，避免上下文无限膨胀。

主要手段：

摘要压缩（Summarization）：当上下文超过阈值时，用 LLM 对历史内容进行摘要，用简短的摘要替换原始内容。

工具结果过滤：对工具调用返回的大量原始数据进行过滤，只保留关键信息。

策略二：持久化上下文（Persisting Context）

持久化是指建立跨会话的信息存储和检索机制，让 Agent 拥有”长期记忆”。

文件存储：最简单的持久化方式。Claude Code 使用 CLAUDE.md，Cursor 使用规则文件。

向量数据库：用于存储和检索语义相关的记忆片段。

自动记忆更新：Agent 在执行过程中自动识别并保存重要信息。

策略三：隔离上下文（Isolating Context）

隔离是指将上下文在不同 Agent 或不同信息类型之间进行分区，避免信息相互干扰。

结构化状态（Schema）：用 Pydantic 模型定义 Agent 的状态结构，而不是把所有信息堆在消息列表里。

多 Agent 隔离：将复杂任务拆分给多个子 Agent，每个子 Agent 只持有自己任务相关的上下文。

Anthropic 的研究表明：多 Agent + 隔离上下文的方案比单 Agent 性能高出 90.2%，主要原因就是 Token 使用效率的提升。

沙箱隔离：HuggingFace 的 CodeAgent 让 Agent 输出代码在沙箱中执行，执行结果（如图片、大型数据结构）存储在沙箱变量中，不直接放入 LLM 上下文，需要时通过变量名引用。

五、四大管理方法

除了三大策略，上下文工程还有四种具体的管理方法：

方法 说明 适用场景
写入（Write） 将重要信息主动写入上下文或外部存储 用户偏好、任务进度、关键发现
选择（Select） 从可用信息中筛选最相关的内容 RAG 检索、工具结果过滤
压缩（Compress） 将长内容摘要为简洁版本 历史对话、长文档、工具输出
隔离（Isolate） 将不同类型信息分区管理 多 Agent 系统、结构化状态

6-上下文工程的实践原则

基于业界实践，上下文工程有几条重要原则：

原则一：信息质量优于数量

更多的上下文不等于更好的结果。一个包含高质量、高相关性信息的精简上下文，往往优于一个充斥着可能有用但实际无关内容的庞大上下文。

原则二：在工具调用边界进行筛选

工具调用是添加筛选逻辑的天然节点。在工具返回结果后立即进行过滤和摘要，比在整个 Agent 轨迹末尾做全局压缩更高效。

原则三：保留错误路径

反直觉但重要：当 Agent 走了弯路时，不要把错误记录从上下文中删除。保留”尝试了 X，但失败了，原因是 Y”这样的记录，可以防止 Agent 重复同样的错误。

原则四：优先监控 Token 消耗

始终追踪 Token 使用情况。及时发现哪些工具调用消耗了大量 Token，是优化上下文工程的基础。

原则五：简化优于复杂

Manus 团队的经验：产品性能的最大飞跃往往来自简化架构、移除不必要的技巧，而不是增加更复杂的上下文管理技术。上下文工程的目标是让 LLM 的工作变得更简单，而非更复杂。

总结

上下文工程是 AI Agent 时代的核心工程方法论，它解决的是”如何在有限的上下文窗口内，为 LLM 提供恰到好处的信息”这一根本问题。

三大核心策略是：筛选（控制 Token 数量）、持久化（建立长期记忆）、隔离（分区管理信息）。

与提示词工程相比，上下文工程更关注系统架构和信息流的动态管理，是构建生产级 AI Agent 的必备技能。
