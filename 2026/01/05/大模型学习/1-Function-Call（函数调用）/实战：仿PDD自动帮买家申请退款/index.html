第四篇：实战：仿PDD自动帮买家申请退款

前言

本文是「大模型学习」系列文章之一，持续更新中。

日常工作中，安排一次多人会议往往需要反复沟通：先问每个人的空闲时间，再找到公共时间段，最后发出邀请。本文用 Spring AI Tool Calling 实现一个智能日程助手，让 AI 自动完成这一系列操作。

需求分析

用户说：”帮我约一下张三和李四，明天下午找个空闲时间开个一小时的需求评审会。”

智能助手需要：

  1. 查询张三、李四和当前用户明天下午的日程
  2. 找到三人都空闲的一小时时间段
  3. 创建会议并发送邀请
  4. 告知用户会议安排结果

整个过程需要调用多个工具，且有先后依赖关系——这正是 Tool Calling 的典型应用场景。

关键设计点

1. 工具调用顺序由模型决定

不需要在代码里写流程控制，模型会根据 description 自动判断调用顺序：先查各人日程 → 找公共空闲 → 创建会议。这正是 Tool Calling 相比传统硬编码流程的优势。

2. 当前用户通过 ToolContext 传入

当前登录用户的信息不应从对话中提取（防止注入攻击），通过 toolContext 安全传入：

3. 并行工具调用优化

查询多人日程时，模型可以并行调用多次 getUserSchedule，而不是串行等待。Spring AI 支持并行 Tool Calling，可以显著减少响应时间：

4. System Prompt 引导行为

通过 System Prompt 明确处理流程和决策规则（如”优先选择较早的时段”），避免模型每次都询问用户确认，提升体验。

总结

这个实战案例展示了 Tool Calling 在日程管理场景中的应用：用自然语言驱动多步骤业务流程。

传统做法需要写复杂的状态机和流程控制代码，而用 Tool Calling，只需要：

  1. 定义好每个工具的 description，让模型知道”什么时候用哪个工具”
  2. 写好 System Prompt 说明处理流程和决策规则
  3. 模型自动编排工具调用顺序，包括并行调用优化

这正是 AI Agent 的雏形——用 LLM 作为”大脑”，工具作为”手脚”，完成复杂的自动化任务。
