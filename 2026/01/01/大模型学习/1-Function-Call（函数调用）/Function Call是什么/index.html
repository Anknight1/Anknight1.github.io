第一篇：Function Call是什么

发表于2026-01-01|更新于2026-04-29|大模型学习1-Function-Call（函数调用）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

大语言模型（LLM）天生是一个”只会说话”的系统——它能理解语言、生成文本，但无法主动获取实时数据、执行代码或操作外部系统。Function Call（函数调用） 正是为了打破这一限制而生的核心机制。

什么是 Function Call？

Function Call，也叫 Tool Calling（工具调用），是让大语言模型能够与外部系统交互的关键技术。其本质是：

模型不直接执行代码，而是输出一个结构化的函数调用请求，由应用程序负责执行并将结果回传给模型。

打个比方：LLM 是一个聪明但”手无寸铁”的顾问，它知道很多知识，但不能查实时天气、不能操作数据库、不能发邮件。Function Call 给了它一双”手”，让它能够委托外部工具完成这些任务。

为什么需要 Function Call？

传统 LLM 存在三大局限：

1. 知识时效性问题
模型的训练数据有截止日期，无法回答”今天北京天气怎样”这类实时问题。通过 Function Call 调用天气 API，模型就能获取最新数据。

2. 无法执行实际操作
用户说”帮我给老板发邮件”，模型只能生成邮件内容，却无法真正发送。Function Call 让模型可以调用邮件发送函数，真正完成任务。

3. 计算精度问题
LLM 做数学计算容易出错，通过调用计算器工具，可以保证结果的准确性。

Function Call 的完整流程

整个过程中，模型只负责”决策”（调用哪个工具、传什么参数），实际执行由应用层完成。

工具定义的结构

每个工具需要向模型描述清楚三件事：

  • name：工具的唯一标识
  • description：告诉模型这个工具能做什么、什么时候用
  • parameters：工具需要哪些输入参数，以 JSON Schema 格式描述

Function Call vs Tool Call

在 Spring AI 1.0.0.M6 版本之后，官方将 Function Calling 更名为 Tool Calling，原因是：

  • 与行业术语保持一致（OpenAI、Anthropic 等均使用 Tool 术语）
  • 新 API 在工具定义和实现之间提供了更好的分离
  • 简化了构造器模式，改进了错误处理

两者在底层原理上完全一致，只是 API 命名发生了变化。如果你在使用旧版 Spring AI，建议尽快迁移到 Tool Calling API。

哪些模型支持 Function Call？

目前主流大模型均已支持工具调用：

模型 支持情况
GPT-4 / GPT-4o ✅ 原生支持，效果最佳
Claude 3.5 Sonnet ✅ 支持，稳定性好
Qwen（通义千问） ✅ 支持
DeepSeek ✅ 支持
Llama 3.1+ ✅ 支持（需要特定格式）
Gemini 1.5 ✅ 支持

总结

Function Call 是大模型从”聊天工具”进化为”智能助手”的关键一步。它的核心思想是：

  1. 模型负责决策：判断是否需要工具、选择哪个工具、生成调用参数
  2. 应用层负责执行：实际调用工具并获取结果
  3. 结果回传模型：让模型整合工具结果生成最终回答

理解了 Function Call 的原理，就理解了现代 AI Agent 的基础。后续文章将深入介绍如何在 Spring AI 中实现工具调用，以及如何设计高质量的工具定义。
