前言

本文是「大模型学习」系列文章之一，持续更新中。

工具调用的效果好不好，70% 取决于工具定义的质量。模型能否正确选择工具、传入正确参数，完全依赖于你写的 description 和参数定义。本文总结工具定义的核心原则和常见陷阱。

核心原则一：description 要清晰、具体、无歧义

description 是模型理解工具的唯一依据，它需要回答三个问题：

  1. 这个工具能做什么？
  2. 什么情况下应该用它？
  3. 有什么限制或注意事项？

核心原则三：工具粒度要合适

太粗粒度：一个工具做太多事，模型难以精确控制。

太细粒度：工具太多，模型选择困难，且增加调用次数。

✅ 合适粒度：一个工具做一件完整的事。

常见陷阱

陷阱一：description 太短

陷阱二：参数名不直观

陷阱三：工具太多导致选择困难

当工具超过 10 个时，模型的选择准确率会明显下降。解决方案：

  1. 工具分组：按业务模块分组，每次只注入相关工具
  2. 意图分类：先用轻量模型判断用户意图，再注入对应工具子集
  3. 工具描述加强：在 description 中明确”不适用场景”

工具定义 Checklist

在定义工具时，可以用这个清单自检：

  • description 是否清楚说明了工具的功能？
  • description 是否说明了适用场景？
  • description 是否说明了限制条件？
  • 每个参数是否有清晰的描述？
  • 参数格式/取值范围是否有说明？
  • 工具名称是否语义化（动词+名词）？
  • 工具粒度是否合适（不太粗也不太细）？
  • 是否与其他工具有功能重叠？
  • 返回值是否包含足够的上下文信息？

总结

好的工具定义是 Function Call 成功的关键。核心要点：

  1. description 要详细：说清楚能做什么、什么时候用、有什么限制
  2. 参数描述要具体：格式、范围、示例一个都不能少
  3. 粒度要合适：一个工具做一件完整的事
  4. 避免重叠：工具之间职责清晰，不要让模型困惑
  5. 工具数量要控制：超过 10 个时考虑动态注入
