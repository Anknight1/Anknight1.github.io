第一篇：MCP技术原理深度解析

发表于2026-01-11|更新于2026-04-29|大模型学习2-MCP（模型上下文协议）

前言

本文是「大模型学习」系列文章之一，持续更新中。

了解 MCP 的使用方式之后，本篇从技术原理层面深入剖析 MCP 的底层机制，帮助你真正理解 MCP 是如何工作的。内容分为三部分：消息协议层（JSON-RPC 2.0）、传输层（三种传输方式）、高级特性（Resources / Prompts / Sampling）。

一、MCP 的协议分层

MCP 协议由两层组成：

消息协议层定义消息的格式和语义（请求、响应、通知），传输协议层定义消息如何在网络或进程间传输。这种分层设计使得 MCP 可以在不同的传输方式上运行，而不需要改变上层的消息格式。

二、JSON-RPC 2.0 消息格式

MCP 使用 JSON-RPC 2.0 作为消息协议，有三种消息类型：

1. 请求（Request）

2. 响应（Response）

3. 通知（Notification）

单向消息，不需要响应（没有 id 字段）：

通知用于服务端主动推送事件，比如工具列表发生变化。

MCP 标准方法列表

MCP 协议定义了一组标准方法：

方法 方向 说明
initialize Client → Server 初始化连接，协商能力
initialized Client → Server 通知初始化完成（通知类型）
tools/list Client → Server 获取工具列表
tools/call Client → Server 调用工具
resources/list Client → Server 获取资源列表
resources/read Client → Server 读取资源内容
prompts/list Client → Server 获取提示模板列表
prompts/get Client → Server 获取特定提示模板
notifications/tools/list_changed Server → Client 工具列表变更通知
ping 双向 心跳检测

错误码规范

错误码 含义
-32700 Parse error（JSON 解析失败）
-32600 Invalid Request（请求格式错误）
-32601 Method not found（方法不存在）
-32602 Invalid params（参数错误）
-32603 Internal error（服务器内部错误）
-32001 Request timeout（请求超时）
-32002 Resource not found（资源不存在）

三、连接建立（三步握手）

MCP 连接建立分为三步：

第一步：Client 发送 initialize 请求

第二步：Server 返回 initialize 响应

第三步：Client 发送 initialized 通知

这是一个通知（没有 id），告诉 Server 客户端已准备好，可以开始正常通信。

能力协商（Capabilities）

握手过程中，Client 和 Server 互相声明自己支持的能力，这叫能力协商。双方只使用对方声明支持的能力，避免调用不支持的功能导致错误。

Server 可以声明的能力包括：tools.listChanged（工具列表变更通知）、resources.subscribe（资源订阅）、prompts.listChanged（提示模板变更通知）、logging（日志功能）、sampling（反向调用 LLM）。

Client 可以声明的能力包括：roots.listChanged（根目录列表变更通知）、sampling（支持被 Server 请求采样）。

五、三种传输方式

MCP 支持三种传输方式，适应不同的部署场景：

stdio 传输

stdio（标准输入/输出）是最简单的传输方式，适用于本地进程通信：

Client 通过 stdin 向 Server 发送 JSON-RPC 消息，Server 通过 stdout 向 Client 返回响应，每条消息以换行符 \n 分隔，Server 的 stderr 用于输出日志（不参与协议通信）。

适用场景：Claude Desktop、Cursor 等本地 AI 工具集成，以及开发调试阶段。

SSE 传输（旧规范）

SSE（Server-Sent Events）是 MCP 早期版本（2024-11-05）使用的远程传输方式：

Client 先通过 GET /sse 建立 SSE 长连接，Server 推送 endpoint 事件告知消息 URL，Client 通过 POST /message 发送请求，Server 通过 SSE 连接推送响应。

SSE 传输存在一些固有缺陷：需要两个连接（SSE 长连接 + POST 请求）、负载均衡困难（SSE 长连接有状态）、部分网络环境不支持 SSE 长连接、断线重连需要额外处理。

Streamable HTTP 传输（新规范，推荐）

Streamable HTTP 是 MCP 2025-03-26 规范引入的新传输方式，解决了 SSE 的问题：

所有通信都通过单一端点 POST /mcp 完成，响应可以是普通 JSON（application/json）或 SSE 流式响应（text/event-stream），Server 根据需要动态选择。这种方式解决了 SSE 的负载均衡和防火墙问题，是目前推荐的远程传输方式。

三种传输方式对比

特性 stdio SSE（旧） Streamable HTTP（新）
适用场景 本地进程 远程服务 远程服务（推荐）
连接数 1个进程 2个（SSE+POST） 1个端点
负载均衡 不适用 困难 容易
防火墙友好 不适用 一般 好
规范版本 所有版本 2024-11-05 2025-03-26+

选择建议：新项目优先使用 Streamable HTTP，需要兼容旧客户端使用 SSE，本地工具集成使用 stdio。

六、MCP 的四大能力

MCP Server 可以向 Client 暴露四种能力：Tools（工具）、Resources（资源）、Prompts（提示模板）、Sampling（采样）。

Resources（资源）

Resources 是 MCP Server 向 Client 暴露的数据内容，可以是文件、数据库记录、API 响应等。与 Tools 不同，Resources 是只读的数据，不执行操作。

Prompts（提示模板）

Prompts 是 MCP Server 提供的可复用提示模板，允许 Server 定义标准化的提示词，Client 可以获取并使用这些模板。典型用途包括代码审查模板、文档生成模板、数据分析模板等。

Sampling（采样）

Sampling 是 MCP 中最特殊的能力——它允许 Server 反向请求 Client 调用 LLM。

有些 MCP Server 需要在工具执行过程中调用 LLM，但 Server 本身不直接持有 LLM 访问权限。通过 Sampling，Server 可以委托 Client 来完成 LLM 调用。典型场景包括：Server 需要对工具返回的数据进行 AI 分析、多步骤 Agent 流程中的中间 LLM 调用等。

MCP 规范特别强调，Sampling 请求应该经过人工审批（Human-in-the-loop），防止恶意 Server 通过 Sampling 绕过用户意图，执行未经授权的 LLM 调用。

四大能力对比

特性 方向 用途 有无副作用
Tools Client → Server 执行操作 有
Resources Client → Server 读取数据 无
Prompts Client → Server 获取提示模板 无
Sampling Server → Client 请求 LLM 调用 有（调用 LLM）

总结

MCP 的技术原理可以概括为：

基于 JSON-RPC 2.0 的消息格式，支持三种消息类型（请求/响应/通知）；通过三步握手建立连接并协商能力；支持 stdio / SSE / Streamable HTTP 三种传输方式，新项目推荐 Streamable HTTP；提供 Tools / Resources / Prompts / Sampling 四种能力类型，其中 Sampling 是唯一的反向调用机制。

理解这些底层机制，有助于在实际开发中更好地设计和调试 MCP 应用。
