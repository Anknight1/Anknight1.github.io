第四篇：实战：开发一个代码审查Skill

前言

本文是「大模型学习」系列文章之一，持续更新中。

前几篇介绍了 Agent Skills 的概念和使用方法。本篇进入实战，从零开始开发一个代码审查 Skill（Code Review Skill），完整演示 Skill 的设计思路、目录结构和 SKILL.md 的编写方法，并对每个设计决策进行深度解析。

一、需求分析

代码审查是一个典型的适合封装成 Skill 的场景：

为什么适合做成 Skill？

首先，这是一个高频重复的任务——开发团队每天都要进行大量 Code Review，每次都要手动输入审查标准非常低效。其次，审查标准需要保持一致性——不同人审查同一段代码可能得出不同结论，Skill 可以固化团队的审查规范。最后，审查流程是可以标准化的——从代码理解到问题识别再到改进建议，有清晰的 SOP。

审查维度设计：

一份好的代码审查应该覆盖以下维度：代码正确性、可读性与命名规范、性能与复杂度、安全风险、测试覆盖度、架构设计合理性。

name 字段的设计原则：

使用小写字母和连字符，简洁明了。code-reviewer 比 CodeReviewer 或 代码审查 更符合 Agent Skills 的命名规范，也更容易在手动触发时输入（/code-reviewer）。

description 字段的设计原则：

description 是 Claude 进行任务路由的核心依据，需要做到三点：说明 Skill 的功能（”对代码进行专业的 Code Review”）；说明覆盖范围（”覆盖代码正确性、可读性…”）；明确触发条件（”当用户提到…时触发”）。

特别是触发条件，要包含用户可能说的各种表达方式，包括中文和英文，正式和口语化的表达。

指令层的设计

指令层采用了四步流水线的结构：代码理解 → 多维度审查 → 综合评估 → 输出报告。

这种结构的优势在于：每一步都有明确的输入和输出，Claude 不会跳步骤；步骤之间有逻辑依赖关系，确保审查的系统性；最后一步明确指向模板文件，保证输出格式的一致性。

权重设计的考量：

这个权重分配反映了代码审查的实际优先级：正确性是最核心的（有 Bug 的代码再漂亮也没用），可读性和性能并列第二，安全问题虽然权重不高但一旦出现就是 Blocker，测试覆盖相对次要。

目录结构的设计逻辑

为什么要把内容分散到多个文件？

这正是渐进式披露的体现。SKILL.md 的正文只包含核心的执行流程（约 1500 tokens），详细的审查清单、安全漏洞模式等放在子目录中。

当 Claude 执行审查时：第一步到第三步只需要 SKILL.md 的指令；第四步输出报告时，才需要加载 templates/review-report.md；如果发现安全问题，才加载 references/security-patterns.md。

这样，一次简单的代码审查可能只需要加载 2000 tokens，而不是把所有文件都塞进上下文。

八、常见问题和优化建议

问题一：Claude 没有自动触发 Skill

原因通常是 description 中的触发词与用户的实际表达不匹配。解决方法是扩充触发词列表，包含更多口语化的表达：

问题二：审查报告格式不一致

原因是 SKILL.md 中对输出格式的描述不够明确。解决方法是在 templates 目录中提供一个完整的示例报告，并在指令中明确说明”使用 templates/review-report.md 中的格式输出”。

问题三：安全问题识别不准确

原因是安全审查标准不够具体。解决方法是在 references/security-patterns.md 中提供详细的安全漏洞模式和代码示例，并在 SKILL.md 中引用：”安全审查时参考 references/security-patterns.md 中的漏洞模式”。

总结

开发一个代码审查 Skill 的核心步骤是：分析需求确定审查维度 → 设计目录结构（SKILL.md + 模板 + 参考资料 + 示例）→ 编写 SKILL.md（元数据层 + 指令层）→ 安装到技能目录 → 验证触发效果。

好的 Skill 设计遵循三个原则：元数据层精准描述触发场景，指令层提供清晰的 SOP，资源层存放不常用的参考资料。这样既保证了 Skill 能被正确触发，又通过渐进式披露机制节省了 Token 消耗。

Skill 的本质是：把人类的专业知识和工作流程，转化为 AI 可以理解和执行的结构化指令。
