前言

本文是「大模型学习」系列文章之一，持续更新中。

MCP Server 暴露了工具和资源，如果不加保护，任何人都可以调用。本篇介绍如何为 MCP Server 添加认证鉴权，保护敏感工具不被未授权访问。

为什么需要认证鉴权

MCP Server 通常会暴露一些敏感操作：

  • 读写文件系统
  • 查询数据库
  • 调用第三方 API（带有 API Key）
  • 执行系统命令

如果没有认证，任何能访问 MCP Server 的客户端都可以执行这些操作，存在严重安全风险。

MCP 规范中的认证方案

MCP 2025-03-26 规范引入了基于 OAuth 2.1 的标准认证方案，主要针对 HTTP 传输（SSE 和 Streamable HTTP）。

认证方案选型建议

场景 推荐方案
本地开发/调试 无认证或 API Key
内网部署 API Key 或 IP 白名单
企业内部服务 JWT Token
对外开放 API OAuth 2.1
多租户 SaaS OAuth 2.1 + 工具级权限

总结

MCP Server 的认证鉴权有多种方案：

  • API Key：最简单，适合内部使用
  • JWT：携带用户信息，适合企业内部
  • OAuth 2.1：MCP 规范标准，适合对外开放
  • IP 白名单：最简粗暴，适合内网隔离

实际项目中，建议至少实现 API Key 认证，防止未授权访问。对于生产环境，推荐使用 JWT 或 OAuth 2.1。
