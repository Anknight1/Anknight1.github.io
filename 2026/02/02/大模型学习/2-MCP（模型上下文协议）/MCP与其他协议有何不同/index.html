第三篇：MCP与其他协议有何不同

发表于2026-02-02|更新于2026-04-29|大模型学习2-MCP（模型上下文协议）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

MCP 出现后，很多人会问：它和 Function Calling 有什么区别？和 A2A 协议又是什么关系？本文从定位、层次、适用场景三个维度彻底厘清这些概念。

MCP vs Function Calling

这是最常见的混淆，一句话说清楚：

Function Calling 是模型的”推理能力”，MCP 是工具的”接入标准”。

本质区别

Function Calling（工具调用）：

  • 属于 LLM 推理层的能力
  • 模型通过训练学会了”将自然语言意图映射为结构化 JSON 调用”
  • 每家模型厂商有自己的格式（OpenAI、Claude、DeepSeek 格式各不同）
  • 解决的问题：模型如何决定调用哪个工具、传什么参数

MCP（模型上下文协议）：

  • 属于 应用层通信协议
  • 基于 JSON-RPC 2.0，规范工具如何暴露自己、如何被调用
  • 跨平台、跨模型的统一标准
  • 解决的问题：工具如何标准化地对外提供服务

协作关系

两者不是替代关系，而是协作关系：

Function Calling 负责”决策”，MCP 负责”执行通道”。

对比表格

维度 Function Calling MCP
本质 LLM 推理能力 通信协议标准
层次 模型层 应用层
提出方 各模型厂商（OpenAI 等） Anthropic（2024）
耦合度 紧耦合（依赖特定模型格式） 松耦合（跨平台）
状态 无状态（每次调用独立） 有状态（持久连接）
工具发现 需手动注册到每个模型 自动发现（tools/list）
适用场景 单次/多次函数调用 工具生态标准化

MCP vs A2A

A2A（Agent-to-Agent） 是 Google 于 2025 年提出的协议，专门解决 AI Agent 之间的协作通信问题。

定位区别

协议 解决的问题 通信对象
MCP AI 模型与外部工具/数据源的连接 模型 ↔ 工具
A2A AI Agent 之间的任务委托与协作 Agent ↔ Agent

场景举例

MCP 的场景：

  • Claude 通过 MCP 读取本地文件
  • GPT-4 通过 MCP 查询数据库
  • AI 助手通过 MCP 调用天气 API

A2A 的场景：

  • 主 Agent 将”写代码”子任务委托给专门的代码 Agent
  • 研究 Agent 将”搜索资料”委托给搜索 Agent
  • 多个专业 Agent 协作完成复杂任务

三者的完整关系

三者是互补关系，共同构成现代 AI 应用的技术栈：

  • Function Calling：模型的”大脑决策”
  • MCP：连接外部世界的”标准接口”
  • A2A：Agent 之间的”协作语言”

MCP vs OpenAPI / REST

有人会问：MCP 和普通的 REST API 有什么区别？为什么不直接用 HTTP 接口？

维度 REST API MCP
设计理念 面向资源（CRUD） 面向能力（工具/资源/提示）
工具发现 需要手动查阅文档 自动发现（tools/list）
AI 友好性 需要额外适配层 原生为 AI 设计
状态管理 无状态 有状态（持久连接）
流式支持 需要额外实现 原生支持 SSE 流式
能力描述 OpenAPI Spec（人读） JSON Schema（机器读）

MCP 的核心优势是AI 原生设计：工具的描述格式、参数 Schema、错误码都是为 LLM 量身定制的，模型可以直接理解和使用，无需额外的适配层。

总结

  • Function Calling = 模型的推理能力，决定”调什么”
  • MCP = 工具的接入标准，规范”怎么调”
  • A2A = Agent 间的协作协议，解决”谁来调”
  • REST API = 通用 Web 接口，不专门为 AI 设计

在实际项目中，三者往往同时使用：模型通过 Function Calling 决策，通过 MCP 调用工具，复杂任务通过 A2A 分发给专业 Agent。
