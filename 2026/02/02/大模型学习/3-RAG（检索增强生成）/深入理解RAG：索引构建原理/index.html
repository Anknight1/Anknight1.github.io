第二篇：深入理解RAG：索引构建原理

发表于2026-02-02|更新于2026-04-29|大模型学习3-RAG（检索增强生成）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

RAG 的完整流程分为两个阶段：离线的索引构建和在线的检索生成。本篇深入介绍这两个阶段的每个环节，帮助你理解 RAG 的底层工作原理。

一、索引构建阶段

索引构建是 RAG 的准备工作——将原始文档转换为可被高效检索的向量索引。

① 文档加载

文档加载器（Document Loader）负责从各种数据源读取内容，统一转换为标准的 Document 对象。

文档类型 加载工具 说明
PDF PDFBox、PyPDF 提取文本，处理多列布局
Word (.docx) Apache POI 提取段落、表格
HTML/网页 Jsoup 提取正文，去除导航栏等
Markdown 直接解析 保留结构信息
数据库 JDBC 将记录转为文档

② 文档清洗

原始文档往往包含大量噪声，需要清洗：去除页眉页脚、去除特殊字符、格式统一、提取元数据（文件名、创建时间、作者、章节标题）。

③ 文档分片（Chunking）

这是索引构建中最关键的步骤，分片质量直接影响检索效果。

为什么需要分片： LLM 有 Context Window 限制；向量相似度搜索需要粒度适中的文本块；太长会包含太多无关信息，太短则缺乏上下文。

常见分片策略：

固定大小分片（Fixed Size）：

按语义边界分片（Semantic）： 按段落、句子等自然边界切分，保持语义完整性。

递归分片（Recursive）： 先按大分隔符（如 \n\n）切分，如果还太长，再按小分隔符（如 \n、.）继续切分。

分片参数参考：

场景 推荐 chunk size 推荐 overlap
通用文档 512 tokens 50 tokens
技术文档/代码 256 tokens 30 tokens
长篇报告 1024 tokens 100 tokens
对话/FAQ 128 tokens 20 tokens

④ 向量化（Embedding）

Embedding 模型将文本转换为高维向量，语义相似的文本在向量空间中距离更近。

常用 Embedding 模型：

模型 维度 特点
text-embedding-3-small（OpenAI） 1536 性价比高
text-embedding-3-large（OpenAI） 3072 效果最好
bge-m3（BAAI） 1024 开源，多语言
text2vec-large-chinese 1024 中文效果好

二、检索生成阶段

当用户提问时，系统进入在线的检索生成阶段：

① 问题向量化

用户的问题需要用同一个 Embedding 模型转换为向量，才能与索引中的向量进行比较。索引时用什么模型，检索时必须用同一个模型，不同模型的向量空间不兼容。

② 向量相似度搜索

相似度度量方式：余弦相似度（最常用，不受向量长度影响）、欧氏距离（受向量长度影响）、点积（计算最快，需要归一化向量）。大多数场景下使用余弦相似度。

Top-K 的选择：K 太小可能遗漏重要信息，K 太大引入噪声，推荐 K=3~5 配合相似度阈值过滤。

③ 结果后处理

重排序（Reranking）： 向量检索基于语义相似度，但不一定是最相关的。重排序模型（Cross-Encoder）可以更精确地评估相关性：

去重： 多个文档片段可能来自同一段落，需要去重处理。

总结

RAG 的完整技术原理分为两个阶段：

索引构建（离线）： 文档加载 → 文档清洗 → 文档分片 → 向量化 → 存储。其中分片策略和 Embedding 模型的选择对最终检索效果影响最大。

检索生成（在线）： 问题向量化 → 相似度搜索 → 结果后处理 → Prompt 构建 → LLM 生成。Prompt 模板的设计和检索参数的调优是影响 RAG 效果的关键因素。
