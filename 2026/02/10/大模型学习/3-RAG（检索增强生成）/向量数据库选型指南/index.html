第九篇：向量数据库选型指南

发表于2026-02-10|更新于2026-04-29|大模型学习3-RAG（检索增强生成）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

向量数据库是 RAG 系统的核心存储组件。市面上有十几种向量数据库，如何选择适合自己场景的？本篇从多个维度进行对比分析。

向量数据库的核心功能

向量数据库需要支持：

  1. 向量存储：存储高维向量（通常 768~3072 维）
  2. 近似最近邻搜索（ANN）：快速找到最相似的向量
  3. 元数据过滤：结合标量条件过滤（如 category = "技术"）
  4. CRUD 操作：增删改查向量数据

主流向量数据库对比

Chroma

定位： 开发者友好的嵌入式向量数据库

优点： 零配置、开箱即用、支持本地持久化、Python/Java 客户端

缺点： 不适合大规模生产（百万级以上数据性能下降）

适用场景： 开发测试、小型项目、原型验证

Milvus

定位： 开源的高性能向量数据库，适合生产环境

优点： 高性能（支持亿级向量）、丰富的索引类型、支持混合检索

缺点： 部署复杂（依赖 etcd、MinIO）、资源消耗较大

适用场景： 大规模生产环境、企业级应用

Pinecone

定位： 全托管的云向量数据库

优点： 全托管、无需运维、高可用、自动扩缩容

缺点： 收费（按存储和查询量计费）、数据在云端（合规问题）

适用场景： 快速上线、不想运维、预算充足

Weaviate

定位： 支持多模态的向量数据库

优点： 支持多模态（文本+图像）、内置 GraphQL API、支持混合检索

缺点： 学习曲线较陡、资源消耗较大

适用场景： 多模态 RAG、需要 GraphQL 接口

pgvector（PostgreSQL 扩展）

定位： 在 PostgreSQL 中添加向量搜索能力

优点： 复用现有 PostgreSQL 基础设施、支持 SQL 查询、事务支持

缺点： 大规模场景性能不如专用向量数据库

适用场景： 已有 PostgreSQL 基础设施、数据量中等（千万级以下）

Qdrant

定位： 高性能 Rust 实现的向量数据库

优点： 性能优秀（Rust 实现）、支持过滤、支持量化压缩

缺点： 社区相对较小

适用场景： 对性能要求高、中大规模场景

选型决策树

其他考虑因素：

  • 已有 PostgreSQL → pgvector
  • 需要多模态 → Weaviate
  • 国内部署（网络问题）→ Milvus 或 Qdrant
  • 快速原型 → Chroma

Spring AI 支持的向量数据库

Spring AI 1.0 支持以下向量数据库：

  • Apache Cassandra
  • Azure AI Search
  • Chroma
  • Elasticsearch
  • GemFire
  • Milvus
  • MongoDB Atlas
  • Neo4j
  • OpenSearch
  • Oracle
  • pgvector
  • Pinecone
  • Qdrant
  • Redis
  • SAP HANA
  • Typesense
  • Weaviate

切换向量数据库只需修改配置，代码无需改动（Spring AI 的抽象层屏蔽了差异）。

总结

向量数据库选型建议：

  • 开发/原型：Chroma（零配置，开箱即用）
  • 已有 PostgreSQL：pgvector（复用基础设施）
  • 生产环境（自托管）：Milvus 或 Qdrant
  • 生产环境（云托管）：Pinecone
  • 多模态场景：Weaviate

不要过早优化，先用 Chroma 跑通流程，再根据实际需求迁移到合适的数据库。
