前言

本文是「大模型学习」系列文章之一，持续更新中。

HyDE（Hypothetical Document Embeddings，假设文档嵌入）是一种巧妙的 RAG 优化技术。它的核心思想是：与其用问题去检索答案，不如先让 LLM 生成一个假设答案，再用假设答案去检索真实文档。

HyDE 的核心思想

问题：问题和答案的向量空间差异

在向量空间中，问题和答案的表示方式往往不同：

这种语义差异导致问题向量和答案文档向量之间的距离较大，检索效果不理想。

HyDE 的解决方案

假设答案和真实答案在语义空间中更接近，因此检索效果更好。

HyDE 的适用场景

适合 HyDE 的场景：

  • 问题和答案的表述差异较大（如问题是疑问句，答案是陈述句）
  • 专业领域问题（医疗、法律、技术文档）
  • 问题较短但答案较长的场景

不适合 HyDE 的场景：

  • 问题本身就很具体（如”Spring AI 的版本号是多少？”）
  • 对延迟敏感的场景（HyDE 需要额外的 LLM 调用）
  • 假设答案可能严重偏离真实答案的场景

HyDE 的局限性

1. 增加延迟

HyDE 需要额外调用一次 LLM 生成假设答案，增加了响应时间。

2. 假设答案可能有偏差

如果 LLM 对某个领域不熟悉，生成的假设答案可能偏离真实答案，导致检索结果更差。

3. 成本增加

额外的 LLM 调用增加了 Token 消耗和费用。

HyDE 效果评估

总结

HyDE 是一种简单但有效的 RAG 优化技术：

  • 核心思想：用假设答案代替问题做检索，缩小问题和答案的语义差距
  • 实现简单：只需在检索前增加一次 LLM 调用
  • 效果显著：在专业领域问答中通常能提升 10-20% 的召回率
  • 代价：增加延迟和成本

建议在对检索质量要求高、对延迟不敏感的场景中使用 HyDE。
