前言

本文是「大模型学习」系列文章之一，持续更新中。

传统 RAG 是一个固定的”检索 → 生成”流程，而模块化 RAG（Modular RAG）将整个流程拆解为多个可插拔的模块，开发者可以灵活组合各个组件，构建适合自己业务场景的 RAG 管道。

本文介绍 Spring AI 和 LangChain4J 两个框架的模块化 RAG 实现，并进行横向对比。

三、两框架横向对比

架构对比

Spring AI 以 Advisor 链为核心，RAG 是 Advisor 的一种，与日志、记忆等 Advisor 平级组合。LangChain4J 以 AiServices + RetrievalAugmentor 为核心，RAG 管道独立于 AI Service 之外。

核心模块对比

功能 Spring AI LangChain4J
问题改写 RewriteQueryTransformer 无内置，需自定义
问题扩展 MultiQueryExpander ExpandingQueryTransformer
对话压缩 CompressionQueryTransformer CompressingQueryTransformer
向量检索 VectorStoreDocumentRetriever EmbeddingStoreContentRetriever
Web 搜索 无内置 WebSearchContentRetriever
SQL 检索 无内置 SqlDatabaseContentRetriever
查询路由 无内置多源路由 LanguageModelQueryRouter（LLM 路由）
Cohere 重排序 CohereReranker ReRankingContentAggregator
文档压缩 LlmDocumentCompressor 无内置
Spring Boot 集成 原生支持 需要额外配置

如何选择

选择 Spring AI 的场景： 已有 Spring Boot 项目、需要与 Spring 生态（Security、Data 等）深度集成、团队熟悉 Spring 开发模式。

选择 LangChain4J 的场景： 需要多数据源智能路由、需要 Web 搜索实时检索、需要更灵活的 RAG 管道控制。

总结

Spring AI 和 LangChain4J 都提供了完善的模块化 RAG 支持，各有侧重：Spring AI 更注重与 Spring 生态的融合，API 设计符合 Spring 风格，QuestionAnswerAdvisor 让简单场景 3 行代码即可完成；LangChain4J 在查询路由和多数据源方面更灵活，特别是 LanguageModelQueryRouter 支持 LLM 智能路由，功能更丰富。
