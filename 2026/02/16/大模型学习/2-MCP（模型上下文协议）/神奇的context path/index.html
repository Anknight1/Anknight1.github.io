第十三篇：神奇的context path

前言

本文是「大模型学习」系列文章之一，持续更新中。

在实际部署中，MCP Server 往往不是独立运行的，而是作为一个模块集成在现有的 Spring Boot 应用中，或者通过 Nginx 反向代理暴露在特定路径下。这时候 context-path 的配置就变得非常重要，稍有不慎就会导致 MCP 连接失败。

什么是 context-path

context-path 是 Spring Boot 应用的根路径前缀。设置后，所有接口都会加上这个前缀：

设置后：

  • 原来的 /sse → 变成 /api/v1/sse
  • 原来的 /message → 变成 /api/v1/message

MCP Server 的默认路径

Spring AI MCP Server 默认暴露两个端点：

端点 路径 说明
SSE 连接 /sse 建立 SSE 长连接
消息接收 /mcp/message 接收客户端消息

（Streamable HTTP 模式下只有 /mcp 一个端点）

问题场景

场景一：设置了 context-path

此时 MCP 端点变为：

  • SSE：http://localhost:8080/my-app/sse
  • 消息：http://localhost:8080/my-app/mcp/message

但如果客户端仍然连接 http://localhost:8080，就会 404。

场景二：Nginx 反向代理添加了路径前缀

外部访问 https://example.com/mcp-server/sse，但 Spring Boot 收到的是 /sse，这种情况通常没问题。

外部访问 https://example.com/mcp-server/sse，Spring Boot 收到的是 /mcp-server/sse，就需要配置 context-path。

总结

context-path 看似简单，但在 MCP 场景下容易踩坑：

  • 客户端 URL 必须包含 context-path（如果设置了的话）
  • Nginx 反向代理时注意路径的映射关系（末尾斜杠很关键）
  • 多 MCP Server 共存时，通过自定义端点路径区分
  • 调试时用 curl 直接访问 SSE 端点，快速定位路径问题

记住一个原则：客户端配置的 URL + /sse = 实际的 SSE 端点路径。
