第二篇：MCP是什么

发表于2026-02-01|更新于2026-04-29|大模型学习2-MCP（模型上下文协议）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

如果说 Function Call 是给大模型装上了”手”，那 MCP（Model Context Protocol） 就是给这双手制定了一套通用的”接口标准”。本文从零开始讲清楚 MCP 是什么、解决了什么问题，以及它与 Function Calling 的本质区别。

MCP 是什么？

MCP（Model Context Protocol，模型上下文协议） 是由 Anthropic 于 2024 年 11 月 25 日正式发布的开放标准协议。

它的核心目标是：统一 AI 模型与外部数据源、工具之间的通信方式。

最形象的比喻是：MCP 就是 AI 领域的 USB-C 接口。

在 USB-C 出现之前，各种设备有各自的充电接口，每换一台设备就要换一根线。USB-C 统一了接口标准，一根线走天下。MCP 做的是同样的事——在它出现之前，每个 AI 应用想接入 GitHub、数据库、文件系统，都要为每个 LLM 单独写一套适配代码；有了 MCP，工具开发者只需开发一次 MCP Server，所有支持 MCP 的 AI 应用都能直接使用。

MCP 解决了什么问题？

问题：工具集成的碎片化

在 MCP 出现之前，AI 工具集成是这样的：

同一个天气工具，要为三个模型写三套代码。模型 API 一更新，三套代码都要改。

解决方案：统一协议

有了 MCP：

工具开发者只需遵循 MCP 规范开发一次，所有支持 MCP 的 AI 应用都能直接调用。

MCP 的四大核心能力

MCP v1.0 定义了四种核心能力类型：

1. Resources（资源）
只读数据流，让模型能读取外部数据。比如读取本地文件、数据库记录、Git 仓库内容。模型可以”看到”这些数据，但不能修改。

2. Tools（工具）
可执行的动作，模型可以主动触发。比如执行 SQL 查询、发送 Slack 消息、调用外部 API。这是 MCP 中最常用的能力，类似于 Function Calling。

3. Prompts（提示模板）
服务器提供的预设指令集，相当于”标准化操作手册”。比如”重构这段代码”、”生成周报”等特定业务场景的 Prompt 模板，可以被 AI 应用直接调用。

4. Sampling（采样）
允许 MCP Server 反向请求 Host 端的 LLM 进行推理。比如日志分析工具读取几万行日志后，可以请求 Host 的 LLM 来总结错误模式，再将结果返回给用户。

MCP 的三层架构

MCP 采用经典的客户端-服务器架构，包含三个核心角色：

MCP Host（宿主）：运行 AI 模型的应用程序，是用户直接交互的界面。典型例子：Claude Desktop、VS Code 的 Cline 插件、Cursor。

MCP Client（客户端）：嵌入在 Host 内部，负责与 MCP Server 建立连接并通信。每个 Client 与一个 Server 保持 1:1 的连接。

MCP Server（服务器）：轻量级程序，通过标准协议暴露特定能力（Resources/Tools/Prompts）。开发者用 SDK 开发，一次开发，多处复用。

MCP vs Function Calling vs Agent

这是最容易混淆的三个概念，一张表说清楚：

对比维度 MCP Function Calling Agent
本质 应用层通信协议 LLM 推理层能力 任务执行系统
提出方 Anthropic（2024） 各模型厂商 学术界/工业界
耦合度 松耦合（跨平台） 紧耦合（依赖特定模型） 可松可紧
解决问题 工具集成标准化 模型调用外部函数 复杂任务自动化

关键区别：

  • Function Calling 是 LLM 的推理能力（将自然语言映射为结构化 JSON 调用），属于模型层
  • MCP 是通信协议（规范工具如何暴露和被调用），属于应用层
  • 两者不是替代关系，而是协作关系：MCP Server 暴露的 Tools，最终还是通过 Function Calling 机制被模型调用

MCP 的通信协议

MCP 底层使用 JSON-RPC 2.0 作为通信协议，支持两种传输方式：

stdio（标准输入/输出）：适合本地进程间通信，Host 启动 MCP Server 作为子进程，通过 stdin/stdout 通信。轻量、无网络开销，Claude Desktop 就是这种方式。

Streamable HTTP：适合远程部署和生产环境，通过 HTTP 协议通信，支持流式响应。MCP 协议 2025-03-26 版本正式推荐此方式（旧版的 HTTP+SSE 已废弃）。

为什么 MCP 这么火？

2024 年底 MCP 发布后，迅速成为 AI 工具生态的事实标准，原因有三：

  1. Anthropic 背书：Claude 的母公司发布，天然获得信任
  2. 生态爆发：GitHub 上已有数千个开源 MCP Server，覆盖文件系统、数据库、浏览器、代码工具等各类场景
  3. 主流支持：OpenAI、Google、微软等主流 AI 平台相继宣布支持 MCP

总结

MCP 的核心价值在于解耦和标准化：

  • 工具开发者：只需开发一次 MCP Server，无需为每个 AI 平台单独适配
  • AI 应用开发者：接入 MCP 协议后，可以直接使用社区海量的 MCP Server
  • 用户：AI 助手能力大幅扩展，可以访问文件、数据库、各类 API

后续文章将深入介绍如何用 Spring AI 开发 MCP Client 和 MCP Server，以及 MCP 的各种高级特性。
