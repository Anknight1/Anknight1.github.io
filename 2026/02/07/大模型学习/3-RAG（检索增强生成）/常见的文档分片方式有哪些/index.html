第六篇：常见的文档分片方式有哪些

发表于2026-02-07|更新于2026-04-29|大模型学习3-RAG（检索增强生成）
|浏览量:

前言

本文是「大模型学习」系列文章之一，持续更新中。

文档分片（Chunking）是 RAG 系统中最关键的步骤之一。分片策略直接影响检索的精度和召回率。本篇介绍常见的分片方式及其适用场景。

为什么分片策略很重要

分片的本质是在精度和上下文之间寻找平衡：

  • 分片太小：每个片段语义不完整，缺乏上下文，检索到的内容无法独立理解
  • 分片太大：包含太多无关信息，稀释相关性，增加 LLM 的 Context 长度

好的分片策略应该让每个片段：

  1. 语义完整，可以独立理解
  2. 粒度适中，不包含过多无关信息
  3. 与相邻片段有适当重叠，避免信息断裂

策略一：固定大小分片（Fixed Size Chunking）

最简单的分片方式，按固定的 token 数或字符数切分。

优点： 实现简单，片段大小可控

缺点： 可能在句子中间切断，破坏语义完整性

适用场景： 对分片质量要求不高的场景，或文档结构不规则时

策略二：递归字符分片（Recursive Character Splitting）

按优先级尝试不同的分隔符，先用大分隔符，如果还太长再用小分隔符：

优点： 尽量保持语义完整性，优先在自然边界处切分

缺点： 片段大小不均匀

适用场景： 通用文档，是最常用的分片策略

策略三：语义分片（Semantic Chunking）

根据语义相似度决定分片边界——当相邻句子的语义差异超过阈值时，在此处切分。

工作原理：

  1. 将文档切分为句子
  2. 计算相邻句子的 Embedding 相似度
  3. 当相似度低于阈值时，在此处切分

优点： 分片边界在语义转折处，语义完整性最好

缺点： 需要额外的 Embedding 计算，成本较高；片段大小差异很大

适用场景： 对检索质量要求高，且预算充足的场景

策略四：按文档结构分片

利用文档的自然结构（标题、段落、章节）进行分片。

优点： 完全保留文档结构，每个片段对应一个完整的章节

缺点： 章节长度差异很大，长章节可能超过 Context 限制

适用场景： 结构化文档（技术文档、教程、报告）

策略五：父子分片（Parent-Child Chunking）

这是一种特殊的分片策略，将文档分为两个粒度：

  • 父片段（Parent Chunk）：较大的片段，提供完整上下文（如整个段落）
  • 子片段（Child Chunk）：较小的片段，用于精确检索（如单个句子）

检索时用子片段找到最相关的内容，但返回给 LLM 的是对应的父片段（包含更多上下文）。

检索时：用子片段”Spring AI 支持哪些 LLM？”匹配到”它支持多种 LLM 提供商…”
返回时：返回整个父片段，提供完整上下文

策略六：滑动窗口分片

在固定大小分片的基础上，增加较大的重叠窗口：

优点： 确保重要信息不会因为恰好在分片边界而被切断

缺点： 存储量增加（重叠部分重复存储）

分片策略选型指南

文档类型 推荐策略 参数建议
通用文档 递归字符分片 512 tokens, 50 overlap
技术文档/教程 按标题结构分片 -
法律/合同文档 语义分片 高阈值
代码文档 按函数/类分片 -
对话/FAQ 固定大小分片 128 tokens
长篇报告 父子分片 父:1024, 子:256

总结

常见的文档分片策略：

  • 固定大小：最简单，适合快速验证
  • 递归字符：最通用，推荐默认选择
  • 语义分片：质量最好，成本较高
  • 按结构分片：适合结构化文档
  • 父子分片：兼顾精度和上下文，适合复杂场景

没有万能的分片策略，需要根据文档类型和业务需求选择合适的方案，并通过实验验证效果。
